# Code Quality Rules For Codex

- При генерации, изменении, рефакторинге и ревью кода в этом workspace используй `clean-code-rules-codex.md` как базовый источник правил по читаемости, простоте и поддерживаемости.
- Для JavaScript-кода дополнительно опирайся на `javascript-best-practices-codex.md`.
- Для TypeScript-кода дополнительно опирайся на `typescript-best-practices-codex.md`.
- Если инструкции этих документов конфликтуют, применяй такой приоритет:
  1. требования пользователя и локальные инструкции workspace;
  2. `clean-code-rules-codex.md`;
  3. профильный документ языка: `javascript-best-practices-codex.md` или `typescript-best-practices-codex.md`;
  4. существующие соглашения конкретного репозитория, если они не противоречат пунктам выше.
- При написании нового кода и изменении существующего кода считай эти документы практиками по умолчанию и не отступай от них без явной причины.
- Не используй inline type imports в сигнатурах и generic-аннотациях вида `import('...').SomeType`. Для типов используй обычный `import type { SomeType } from '...'` вверху файла.
- В TypeScript-классах не используй `public` accessor по умолчанию, если он не добавляет ясности. Если member и так публичный по умолчанию, предпочитай запись без `public`.
- Не помечай функции и методы как `async`, если внутри нет реальной асинхронной работы. Не используй `await` для синхронных вызовов только ради единообразия сигнатур или "на будущее". Если вызов синхронный, оставляй его синхронным до появления реальной необходимости в асинхронности.
- Если булево условие выражает важную контрактную, доменную или инфраструктурную семантику, предпочитай выносить его в именованную boolean-переменную с `is/has/can/should`-префиксом, когда это делает намерение кода яснее. Не выноси в отдельную переменную тривиальные проверки, если новое имя не добавляет читаемости.
- Если у сервиса или модуля рядом уже есть `types.ts`, связанные с этим сервисом типы и интерфейсы держи именно в этих файлах, а не размазывай по соседним файлам, helper-модулям или consumer-файлам.
- `types.ts` используй для внутренних service-level и module-level типов, `contracts.ts` для более явных публичных контрактов входа/выхода и explainable result shape.
- Экспортируй любой символ модуля только тогда, когда действительно хочешь поддерживать его как отдельную внешнюю точку использования.
- Это правило относится ко всем видам символов: функциям, типам, интерфейсам, классам, константам, enum-ам, factory, builder, utility и другим сущностям.
- Если символ не используется вне файла и не является частью осознанного public API модуля, не делай его `export`.

## Repo-Specific Rules Moved From AGENTS.md

- Не использовать `any`, кроме действительно крайних случаев, и всегда явно понимать границу небезопасного значения.
- Для нового кода сохранять читаемый и предсказуемый flow ответственности; не смешивать transport, business logic и data access в одном месте.
- Не читать `process.env` напрямую вне config layer; использовать только validated config modules.
- Не формировать обычные API response вручную, если проектный flow допускает `Resource` и `BaseController.execute(...)`.
- Если правило относится к качеству и стилю кода, а не к архитектурной карте проекта, оно должно жить в этом файле, а не в `AGENTS.md`.

# Unit-Aware Naming Rules

- Все переменные, поля, DTO fields, config values и contract properties, которые выражают величину в конкретной единице измерения, обязаны явно отражать эту единицу в названии.
- Если значение измеряется во времени, процентах, барах, свечах, тиках, объёме, количестве или другой измеримой единице, не используй нейтральные имена без unit semantics.
- Не называй такие значения просто `time`, `duration`, `value`, `change`, `interval`, `amount`, если единица измерения неочевидна из самого имени.
- Предпочитай имена вида `durationMs`, `ttlSec`, `lookbackHours`, `changePct`, `ratioBps`, `candlesCount`, `tickSize`, `volumeUsd`, `startTimeMs`, `openTimeUnixSec`.
- Для времени и длительностей используй явные суффиксы или эквивалентные обозначения единицы: `ms`, `sec`, `min`, `hours`, `days`.
- Для процентных и относительных величин используй явные обозначения: `pct`, `percent`, `ratio`, `bps`, `multiplier`.
- Для рыночных и расчётных величин используй единицы или семантически явные обозначения там, где это применимо: `price`, `tickSize`, `ticks`, `volume`, `qty`, `count`, `bars`, `candles`.
- Исключение: общепринятые pagination и aggregate имена `offset`, `limit`, `total`, `count` не нужно переименовывать только ради unit-aware naming.
- Если единица измерения нестандартная или доменно-специфичная, она всё равно должна быть читаема из имени, а не оставаться только в комментарии или в голове автора.

## Backend Contract First For UI Integrations

- Для frontend/backend интеграций backend API contract считается источником правды для UI.
- Не подгоняй backend response shape под текущие ожидания UI только ради удобства фронтенда.
- Backend не должен адаптировать доменные данные под presentation-слой UI.
- Backend не должен менять единицы времени, display-oriented naming, sorting, grouping, chart-specific shape или другие presentation-specific преобразования только потому, что так сейчас удобнее конкретному экрану.
- Если frontend должен работать с новым или изменившимся backend contract, адаптацию выполняй на стороне UI: в `src/api`, page service, mapping-слое или ближайшей frontend-нормализации данных.
- Если UI ожидает другой формат данных, именно UI должен маппить backend response под свои компоненты, графики, таблицы и локальные display-модели.
- Backend меняет contract только тогда, когда это действительно доменное или API-level требование, а не workaround под текущую реализацию UI.
- При расхождении между текущим UI и backend contract исправляй UI так, чтобы он отражал фактические данные и ограничения backend, а не наоборот.
